# Karpathy 的 AutoResearch：当 AI 开始自主做研究，这意味着什么？

> **Deep Dive** · 2026年3月 · 约 5000 字

---

## 引言：一夜之间，700 次实验

2026年3月7日深夜，Andrej Karpathy 在 X 上发了一条推文，配上一张实验日志的截图。没有华丽的标题，没有预热的 waitlist，只有一个朴素的 GitHub 链接和一句话："给 AI 一个 GPU，让它自己做研究。"

两天之内，这条推文获得了 **860万次浏览**。那个 GitHub 仓库——autoresearch——斩获 **42,000 颗 Star**。MIT 协议，630 行 Python，没有任何外部依赖，除了 PyTorch。

Karpathy 自己的测试结果是：两天运行了 700 次实验，发现了 20 个优化点，在更大规模的模型上实现了 **11% 的速度提升**。而他本人在此期间睡了两觉。

这不是一个炫技项目。这是一场关于"研究本身能否被自动化"的严肃实验。而它的结论，可能比我们想象的更令人不安。

---

## 一、背景：Karpathy 的研究哲学

在理解 AutoResearch 之前，有必要了解 Karpathy 一贯的思维方式。

作为前 OpenAI 核心成员、特斯拉 AI 总监，Karpathy 以"极简主义"著称。他的 [nanoGPT](https://github.com/karpathy/nanoGPT) 用不到 300 行代码复现了 GPT-2，他的 [llm.c](https://github.com/karpathy/llm.c) 直接用 C 语言重写了训练流程。他的信条可以概括为：**复杂性是研究的敌人，能跑起来的简单系统胜过跑不动的复杂系统**。

AutoResearch 延续了这个哲学，但加入了一个新维度：**如果实验本身可以被委托给 AI，研究者的角色会变成什么？**

这个问题在 2026 年初变得格外紧迫。Claude 3.7、GPT-5 等模型在代码生成和长上下文推理上的能力已经达到了一个临界点——它们不只是能写代码，还能**理解实验结果，做出有依据的假设，并修改策略**。AutoResearch 就是在这个时间点上的一次实战检验。

---

## 二、系统架构：三个文件的契约

AutoResearch 的核心是一个极为简洁的三文件架构，Karpathy 称之为"契约"（contract）：

### 文件一：`prepare.py`（不可修改）

这是系统的地基，**人类和 AI 都不能动它**。它负责：

- 下载训练数据
- 训练 tokenizer
- **定义评估指标**：`val_bpb`（validation bits per byte）

这个文件的不可修改性是整个系统的关键设计决策。它确保了每次实验的评估基准完全一致，无论 AI 如何魔改训练逻辑，都用同一把尺子量。

### 文件二：`train.py`（AI 的游乐场）

这是 AI 可以**任意改写**的文件，包含：

- GPT 模型定义
- 优化器配置（初始设置为 Muon + AdamW）
- 训练循环

AI 可以修改任何东西：注意力机制、层数、学习率调度、批次大小、甚至整个架构。唯一的约束是：**不能安装新的 Python 包**（只能用已声明的依赖）。这个约束防止 AI 走旁门左道，比如直接调用某个已经训练好的外部模型。

### 文件三：`program.md`（人类的意图）

这是人类研究者与 AI 沟通的唯一接口。用自然语言写的 Markdown 文件，包含：

- 研究方向和目标
- 约束条件
- 停止标准
- 允许/禁止的探索范围

例如："专注于注意力机制的改进，不要改变数据处理流程，如果 val_bpb 连续 10 次实验没有改善则停止。"

这个设计非常微妙：它不是让人类写代码，而是让人类**写意图**。AI 负责将意图翻译成实验。

## 三、核心机制：5 分钟的时钟

AutoResearch 最反直觉的设计是它的时间预算：**每次实验恰好运行 5 分钟，不多不少**。

这个约束解决了一个困扰 ML 研究者多年的问题：**实验结果不可比**。当一个实验跑了 10 小时，另一个跑了 2 小时，你怎么公平评估哪个架构更好？

固定时间预算意味着：
- 每小时约 12 次实验
- 一夜（8小时）约 96 次实验
- 所有实验在**相同计算量**下直接可比

配合 `val_bpb` 这个与词汇表大小无关的指标（它测量的是每字节原始 UTF-8 文本的压缩效率），即使 AI 改变了 tokenization 策略，实验结果依然可以横向对比。

整个循环如下：

```
[修改 train.py] → [训练 5 分钟] → [评估 val_bpb] → [保留/丢弃改动] → [记录结果] → [下一次迭代]
```

AI 在每次迭代开始时会读取所有历史实验记录，基于这些信息决定下一步尝试什么。这是一个真正的**有记忆的迭代优化系统**，而不是随机搜索。

## 四、真实结果：数字说话

### Karpathy 自己的实验

Karpathy 让系统运行了两天，结果令他本人都感到意外：

- **700 次实验**自主运行完成
- 发现了 **20 个优化点**，包括一个他自己写代码时犯的错误——QK-Norm 实现中遗漏了一个标量乘数
- 在较大规模模型上实现 **11% 的速度提升**

最后一点尤其有趣。AI 不只是在做网格搜索，它发现了人类研究者忽略的 bug，并通过自主实验验证了修复方案。

### Shopify CEO 的复刻

Tobi Lütke（Shopify 联合创始人 & CEO）在看到项目后立刻着手复刻，将其应用于内部模型优化：

- 一夜运行 **37 次实验**
- 模型质量和速度综合提升 **19%**
- 随后扩展到内部模板引擎，93 次自动提交后，渲染速度提升 **53%**

这是 AutoResearch 从 ML 研究工具走向通用工程自动化工具的第一个信号。

### 扩展到多 GPU 集群

研究团队随后通过 SkyPilot 将系统扩展到 **16 块 GPU**（13 块 H100 + 3 块 H200），结果：

- 8 小时完成 **910 次实验**（单 GPU 需要约 72 小时）
- 吞吐量提升 **9 倍**
- `val_bpb` 从 1.003 降至 0.974，改善 **2.87%**
- AI 自发学会了用 H200 做验证，用 H100 做初筛——利用了不同硬件的性能差异

这最后一点没有人告诉 AI 怎么做。它自己"想"到了。

## 五、与 AutoML 的根本区别

AutoResearch 发布后，学术界出现了一波质疑声：这不就是 AutoML 吗？Google 的 Neural Architecture Search（NAS）、Microsoft 的 AutoML 工具早就做过类似的事情，为什么现在要大惊小怪？

Karpathy 的回答相当直接，而且有说服力：

**传统 AutoML 是"选择"，AutoResearch 是"发明"。**

传统 NAS 的工作原理是在预定义的搜索空间中寻找最优配置——它可以选择"用 6 层还是 12 层注意力"，但它无法决定"注意力机制本身需要改变"。它的搜索边界是人类事先划定的。

AutoResearch 中的 AI 可以**重写整个训练逻辑**。它不只是调参，它可以换算法、换架构、换优化器。它的搜索边界是"任何合法的 Python 代码"。

更重要的是，传统 AutoML 没有理解能力——它不知道为什么一个配置更好，只知道指标更高。AutoResearch 的 AI 在每次实验后会生成对结果的解释和下一步的假设，它在**模拟科学推理**，而不是盲目搜索。

这个区别在实践中体现得很清楚：在 Karpathy 的实验中，AI 不只是找到了"更好的超参数"，它还诊断出了代码中的 bug——这是任何搜索算法都做不到的。

## 六、技术争议：验证集会被"污染"吗？

最严肃的技术批评来自研究者 @alexisthual，他提出了一个经典的 ML 问题：

> "当你跑了几百次实验，每次都用同一个验证集来决定保留还是丢弃改动，你最终保留的到底是真正的改进，还是对这个特定验证集的过拟合？"

这是一个真实的风险，叫做**验证集泄漏**（validation set leakage）。简单来说：如果你的改进是因为恰好契合了验证集的特点，而不是真正提升了模型能力，那么这些"改进"在新数据上可能一无是处。

Karpathy 的回应是承认这是一个需要认真对待的问题，并提出了几个缓解策略：

1. `val_bpb` 评估的数据量（~2090万 token）足够大，难以过拟合
2. 最终结果需要在保留的测试集上验证（与验证集完全分离）
3. 可以定期轮换验证集来检测过拟合

但这个问题并未完全解决。在更极端的场景下（比如跑了 10,000 次实验），验证集泄漏的风险会显著上升。这也许是 AutoResearch 在未来版本中最需要解决的问题之一。

## 七、规模效应：从一个 PhD 到一个研究社区

Karpathy 在项目说明中写了一句话，值得反复品味：

> "目标不是模拟一个 PhD 学生，而是模拟一个由他们组成的研究社区。"

这句话揭示了 AutoResearch 的真正野心。当前的系统是单线程的——一个 AI agent 顺序运行实验。但 Karpathy 的构想是未来版本将支持**并行、协作的 agent 群**，类似于 SETI@home 项目（那个让全球家用电脑共同搜索外星信号的分布式计算项目）。

设想这样一个场景：

- Agent A 专门探索注意力机制的变体
- Agent B 专门测试不同的优化器组合
- Agent C 专门研究数据增强策略
- 所有 agent 共享实验记忆，互相参考结果
- 人类研究者只需要在 `program.md` 中更新研究方向

在这个图景下，一个研究机构可以在一夜之间完成以往需要数月的实验探索。更重要的是，这些探索不受人类研究者的认知偏见限制——AI 会测试那些"看起来不可能有用"但实际上可能奏效的想法。

## 八、更大的图景：研究的自动化意味着什么？

AutoResearch 引发的最深刻的问题不是技术性的，而是认识论的：**如果实验可以自动化，科学研究的本质是什么？**

传统上，研究者的价值在于三个层面：
1. **提出问题**：什么值得研究？
2. **设计实验**：如何检验假设？
3. **解释结果**：结果意味着什么？

AutoResearch 显然已经接管了第 2 层，并在某种程度上替代了部分第 3 层（AI 会生成实验结果的解释）。剩下的问题是：第 1 层和第 3 层的深度解释，AI 何时能够胜任？

乐观的看法是：AutoResearch 解放了研究者，让他们专注于更高层次的问题——研究方向的选择、结果的深度解读、跨领域的联系。就像计算器不消灭数学家，而是让数学家能够探索更复杂的数学。

悲观的看法是：这条路走到尽头，可能是 AI 自主提出研究问题、设计实验、解释结果、撰写论文——人类研究者的角色被缩减为"审稿人"，然后甚至"审稿人"也被 AI 取代。

Karpathy 本人对这个问题持一种谨慎的乐观：他相信人类的直觉和对"什么重要"的判断在相当长的时间内仍是不可替代的。但他也在 X 上写道：

> "所有 LLM 前沿实验室都会这样做。这是最终 Boss 战。"

## 九、对产业界的影响

AutoResearch 不只是一个学术项目。它的 MIT 协议和极简架构意味着任何公司都可以立刻将其应用于自己的场景。

**对 AI 实验室**：加速模型优化的速度，同时减少人力成本。一个工程师配合 AutoResearch，可能产出比一个团队更多的有效实验。

**对大型科技公司**：Shopify 的案例已经证明，这个框架可以迁移到非 ML 场景——任何有明确评估指标的代码优化任务都是潜在的应用场景。

**对初创公司**：计算资源受限的团队可以通过更智能的自动化实验，在单卡 GPU 上完成以前需要大型集群的探索工作。

**对高校研究者**：研究节奏可能大幅加速，但这也意味着"慢慢做一个长期实验"的研究风格会面临竞争压力。

## 十、如何上手

对于想要亲自体验 AutoResearch 的读者，以下是基本步骤：

**环境要求**

- 一块 NVIDIA GPU（社区已有 RTX 卡的适配版本，M系列 Mac 的版本也在开发中）
- Python + PyTorch 环境
- 一个外部 LLM API（Claude、GPT 均可用于驱动 agent）

**基本流程**

```bash
git clone https://github.com/karpathy/autoresearch
cd autoresearch
# 配置你的 LLM API key
# 编写 program.md，描述你的研究目标
python run.py
# 然后……去睡觉
```

**建议的第一个实验**

在 `program.md` 中写一个保守的目标，比如"探索学习率调度策略的改进，不改变模型架构，运行 20 次实验后停止"。这样可以快速理解系统行为，避免因过于开放的搜索空间导致结果难以解释。

## 结语：我们正在目睹什么？

AutoResearch 在某种意义上是 AI 发展的一个递归奇点的预演：**AI 在改进 AI 的训练方式**。

这不是第一次有人尝试这个方向，但它是第一次以如此简洁、开放、可复现的方式实现，并引发了如此广泛的共鸣。630 行代码，一块 GPU，一夜，700 次实验。

更重要的是，它改变了一个认知：研究者曾经相信"真正的研究需要人类直觉"是不可撼动的。AutoResearch 没有证伪这个信念，但它在边界上打了一个楔子——**某些类型的直觉，AI 正在学会模拟**。

接下来的问题是：这个楔子会停在哪里？

没有人知道。但可以确定的是，那些现在就开始思考这个问题的研究者和工程师，会比那些等待答案揭晓的人更有准备。
