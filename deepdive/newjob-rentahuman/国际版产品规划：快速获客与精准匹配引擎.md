# 国际版产品规划：快速获客与精准匹配引擎

> **核心命题**：RentAHuman 用 3 个月积累了 64 万注册用户，但只有 83 个可见的执行者、$2 万 MRR——这是增长和转化同时失败的典型案例。国际版的目标不是更多注册，而是**更快速地积累真实可调度的执行者池，以及精准到分钟级的任务匹配**。
>
> **核心差异化**：不做"注册驱动"的数字，做"可调度密度"驱动的增长。
>
> **文档类型**：国际版产品规划（增长与匹配架构）
> **撰写日期**：2026-04-30

## 一、重新定义"用户获取"的度量单位

现有平台（包括 RentAHuman）的增长误区：把**注册量**当成北极星指标。

注册量是虚荣指标。一个任务发布方真正关心的只有一件事：**当 Agent 发出任务请求，多少分钟内能有合适的人接单**。这个指标叫做「平均可调度响应时间」（MDRT, Mean Dispatchable Response Time），才是这类平台的生死线。

国际版的增长策略完全围绕一个问题展开：**如何在任意城市最快速度内，建立足够密度的「随时可调度执行者池」**？

**新的度量框架**：

| 传统指标（错误）| 正确替代指标 |
|--------------|------------|
| 注册用户数 | 可调度执行者密度（每城市 / 每技能类别）|
| 任务发布数 | 任务到接单的平均时长（MDRT）|
| GMV 总量 | 执行者月活跃率（MAR）|
| DAU | 任务完成率（Task Completion Rate）|

**目标 MDRT**：
- 高频简单任务（拍照、取件）：< 15 分钟
- 中频执行任务（访谈、核验）：< 2 小时
- 低频专业任务（法律见证、实验室采样）：< 24 小时

## 二、供给侧（执行者）快速获客：三条并行通道

### 通道 1：存量池激活——不建新池，接入已有池

**核心思路**：全球已有数千万通过 KYC 认证的零工在各大平台注册，最快的供给获取不是招募新人，而是让这些人变成「可被 Agent 调用的节点」。

**具体执行**：

**A. API 桥接现有平台**
与 TaskRabbit、Fiverr、Thumbtack、Upwork 的注册执行者导流合作。核心价值主张：「你在原平台的简历/评分自动同步，多一个被 AI 雇主选中的机会，不需要重新证明自己。」技术上用 OAuth 授权，导入他们的已有技能标签和历史评分，降低执行者的注册摩擦到接近零。

**B. 专属 Telegram Bot / WhatsApp Bot 入驻**
不强迫执行者下载新 APP：在他们已经活跃的 IM 工具里完成全部流程。

```
执行者操作流 (Telegram):
/start → 语言选择 → 位置授权 → 技能选择（多选）→ 时薪设置
→ 验证码实名 → ✅ 上线，开始接收任务推送

全程 < 4 分钟，无需下载任何 APP
```

Telegram 每条消息对商业使用免费，通知成本接近零，全球覆盖率高（超 9 亿 MAU），在东南亚、中东、拉美等新兴市场尤其强势。WhatsApp 作为西欧和印度的补充通道。

**C. 社区嵌入式招募**
在 Reddit（r/beermoney、r/slavelabour、r/forhire）、Discord 服务器、Facebook 零工群组投放精准招募帖，核心文案不说「注册新平台」，而说「AI 雇主需要你的帮助，你的技能被算法直接定价，不用竞标」。这个叙事对已经在各大平台疲于竞价的零工有强吸引力。

### 通道 2：城市飞轮——先集中再扩散

**最大错误：同时铺 100 个城市，每个城市都没有密度。**

正确策略：每次只激活一个城市，把密度做到 MDRT < 15 分钟，再开下一个城市。城市选择标准：

```
城市优先级评分 = 
    零工经济活跃度（Gig GDP 密度）× 0.4
  + 宽带 / 智能手机普及率 × 0.2
  + AI Agent 开发者社区规模 × 0.3
  + 法律友好度（灵活用工监管宽松）× 0.1
```

**首批目标城市**（按评分排序）：伦敦 → 新加坡 → 旧金山 → 迪拜 → 班加罗尔 → 拉各斯

每个城市设置一个「城市联络人」（City Captain）——本地有一定社区影响力的人，负责招募首批 50-100 名种子执行者。City Captain 提成机制：其招募的执行者完成的每笔任务，City Captain 永久获得 2% 分润。

### 通道 3：执行者自传播飞轮

**推荐裂变设计**：
- 执行者推荐新执行者：被推荐人完成首单后，推荐人获得 $10 奖励
- 执行者完成任务后，系统自动生成可分享的「收入记录卡」（带任务描述和收入金额），一键分享到 Twitter/X、Telegram、WhatsApp
- 每张分享卡带专属邀请链接，追踪转化

**执行者侧的病毒系数目标 K > 1.2**（即每个老执行者平均带来 1.2 个新执行者）。一旦达到这个阈值，供给侧进入自增长状态。

需求侧的获客路径更简单，因为这类平台天然具有开发者传播属性：

```

    → 同事复用
    → 汇报到技术负责人
    → 企业购买专业版 API Key
```

**具体行动**：

公开显示当前正在执行的匿名任务列表（去掉隐私信息），包括任务类型、城市、当前状态、已完成数量。这个页面本身就是最有说服力的产品证明，也是社交媒体传播的素材。

## 四、精准匹配引擎设计

这是与 RentAHuman 拉开差距的核心技术。RentAHuman 目前的匹配方式极为原始：AI 浏览人类列表，靠关键词和位置筛选。这在任务量小时勉强够用，但一旦规模扩大，这个方式会让匹配质量和速度双双崩溃。

### 4.1 架构总览

```
输入：Agent 发出的任务请求
         ↓
┌────────────────────────────────────────────────────┐
│                  匹配引擎 (三阶段)                  │
│                                                    │
│  阶段 1: 召回（毫秒级）                             │
│  向量检索 → 地理过滤 → 可用性过滤 → 候选集(20-50人)  │
│                        ↓                          │
│  阶段 2: 排序（<100ms）                             │
│  多维度评分模型 → Top 5 候选                        │
│                        ↓                          │
│  阶段 3: 确认（<1s）                                │
│  实时可用性确认 → 派单 → Fallback 链               │
└────────────────────────────────────────────────────┘
         ↓
输出：目标执行者 + 预计接单时间（ETA）
```

### 4.2 阶段 1：向量召回（毫秒级响应）

**任务语义向量化**

传统匹配靠关键词，任务说「帮我在 Shibuya 拍一张便利店货架照片」，关键词提取是「拍照」「Shibuya」「便利店」。向量匹配能理解：这个任务需要的能力是「摄影」+「日本东京本地」+「无门槛」，在执行者库里直接找到语义最近的人。

```python
# 任务嵌入
task_vector = embed_model.encode(
    f"任务: {task.description} | 位置: {task.location} | 
      预算: {task.budget} | 截止: {task.deadline_minutes}min"
)

# 向量检索（使用 pgvector 或 Pinecone）
candidates = vector_db.search(
    query=task_vector,
    filters={
        "location_radius_km": 10,
        "is_available": True,
        "min_rating": 4.0,
        "languages": task.required_languages
    },
    top_k=50
)
```

**执行者 Profile 向量化**

执行者 Profile 不是静态文档，而是动态更新的多维向量：
- 技能向量（从历史任务描述中提取，而非自我填写）
- 位置向量（实时 GPS，而非注册地址）
- 可信度向量（评分、完成率、争议率）
- 时序向量（活跃时段规律，预测当前是否能接单）

关键创新：**技能标签不靠执行者自填，靠历史任务文本反推**——一个人完成过 50 个「餐厅实地评测」类任务，系统自动给他标注「餐饮调研」「实地采样」「图文记录」技能，精准度远超自填标签。

### 4.3 阶段 2：多维度排序评分

召回的 50 名候选者，通过以下权重模型排序，选出 Top 5 发送派单：

```
最终得分 = 
    语义匹配度      × 0.30  (任务描述 ↔ 历史任务向量相似度)
  + 地理距离得分    × 0.25  (距任务地点的步行/驾车时间)
  + 实时可用性     × 0.20  (当前在线 + 历史该时段接单率)
  + 执行者质量     × 0.15  (综合评分 × 完成率 × 响应速度)
  + 价格匹配度     × 0.10  (执行者时薪 vs 任务预算的吻合度)
```

**动态权重调整**：不同任务类型的权重不同。时间敏感型任务（截止 < 30 分钟）提高「地理距离」和「实时可用性」权重；专业类任务（科研采样、法律见证）提高「语义匹配度」权重。

### 4.4 阶段 3：实时确认与 Fallback 链

**派单策略：并发询问，不等待**

不采用「派给第一名，等待 5 分钟，无响应再派给第二名」的串行方式（这会让 MDRT 爆炸）。

采用**赛马派单（Race Model）**：

```
同时向 Top 3 执行者推送任务
  ↓
30 秒内第一个接受者获得任务锁
  ↓
其他两人收到"任务已被接单"通知
  ↓
若 30 秒无人接受 → 自动扩展到 Top 10 → 重复
  ↓
若 2 分钟无人接受 → 提高任务奖励 10% → 重发
  ↓
若 10 分钟无人接受 → 通知 Agent "当前区域无可用执行者"
                   + 提供 ETA（基于历史数据预测下一个可用时间）
```

**Fallback 链设计**：任务永远不会静默失败。Agent 在调用 `create_task` 时，必须获得确定性的响应：要么「已匹配，执行者 X，ETA Y 分钟」，要么「暂无执行者，建议时间窗口 Z」。

### 4.5 实时可用性信号系统

这是匹配精准度的关键——知道一个人「现在是否真的有空」比知道他「有这个技能」更重要。

**可用性信号来源**（优先级从高到低）：

| 信号 | 更新频率 | 可信度 |
|------|---------|--------|
| 执行者主动设置「在线」状态 | 手动 | 高（但懒得更新）|
| 最近 30 分钟内有过 App/Bot 活动 | 实时 | 中高 |
| 历史接单时间模式（该时段接单率）| 每日更新 | 中 |
| 当前任务状态（执行中 → 不可用）| 实时 | 高 |
| 地理位置变动速率（快速移动 → 可能在通勤）| 5 分钟更新 | 中 |

综合以上信号，每个执行者实时维护一个「可用性概率分」（0-100），匹配引擎用这个分数而非布尔值「在线/离线」进行筛选。

## 五、执行者侧体验：最快上线 & 最低摩擦派单

### 5.1 执行者上线流程（目标 < 5 分钟）

**触点：Telegram Bot（首选）或 Web**

```
Step 1: /start → 选择语言（30 语言自动识别）
Step 2: 位置确认（一键授权，精确到区）
Step 3: 技能选择（预定义标签，多选，可搜索）
Step 4: 时薪设置（参考当地市场价建议值）
Step 5: 手机号验证（短信 OTP）
Step 6: 照片上传（头像 + 可选证件照，提升可信度加权）
→ 上线完成，立即开始接收任务推送
```

**无需：**简历、学历、背景调查、视频面试。首批任务系统限定低风险类型（拍照、问卷），随着完成数量和评分积累，自动解锁更高价值任务类型。「成长解锁」模型比「事前审查」模型冷启动快 10 倍。

### 5.2 任务推送体验

执行者收到的派单通知包含：

```
🤖 AI 任务来了！
━━━━━━━━━━━━━━
📍 地点: Shibuya Crossing 步行 8 分钟
📋 内容: 在 7-Eleven 拍摄 3 张货架照片
         （指定货架位置已附图）
💰 报酬: $12 USDC（完成后 5 分钟到账）
⏱️ 截止: 45 分钟内完成
━━━━━━━━━━━━━━
[✅ 接单] [❌ 跳过]
```

**关键设计决策**：
- 报酬显示以 USDC 标注（国际版），强调「5 分钟到账」——这是传统平台无法提供的体验，是最强的执行者留存钩子
- 任务说明足够详细，执行者不需要与 AI 对话确认——指令预处理是产品的核心工作，由平台在 Agent 发单时自动结构化

## 六、支付基础设施：即时结算是护城河

国际版最大的竞争优势之一：**执行者完成任务 5 分钟内收到 USDC**。传统平台（TaskRabbit 7 天、Fiverr 14 天）的结算周期是历史遗留问题，这是执行者最大的痛点之一。

### 6.1 支付架构

```
企业客户（委托方）
    ↓ 预授权锁定 USDC（任务发起时）
任务托管智能合约（escrow）
    ↓ 验证通过后自动释放
执行者钱包（USDC / Circle）
    ↓ 选择性提现
法币提现：Stripe Ramp / MoonPay（兑换成本地法币）
```

**为什么用 USDC 而非直接法币**：
- 即时跨境结算，无 SWIFT 延迟
- 任务托管无需银行参与（智能合约自动执行）
- 对执行者来说是「看得见的」：链上余额实时确认
- 对企业来说：锁定资金（预授权），任务不完成不支出

**法币通道**（给不熟悉加密的用户）：
- 企业端：支持 Stripe 企业付款（USD/EUR/GBP）→ 平台自动转为 USDC 锁仓
- 执行者端：支持 MoonPay / Stripe Ramp 法币提现（手续费 1-2%）
- 完全透明：执行者可以永远不碰加密，感受到的只是「完成任务→5分钟到账」

## 七、增长飞轮总设计

三个飞轮联动，形成自增强的增长系统：

```
         [执行者飞轮]
         更多可调度执行者
               ↑
    执行者分润推荐 ←→ 更快的 MDRT
               ↑              ↓
    [平台核心]        更多 Agent 发单
    MDRT < 15min             ↓
               ↓       [任务飞轮]
    更好的匹配数据    更多历史任务数据
               ↓              ↓
    [算法飞轮]        执行者技能向量更准
    匹配精度持续提升 ←← 任务完成数据回流
```

**关键飞轮触发点**：每个城市在执行者达到 **150 名可调度** 时，MDRT 会从「小时级」跌落到「分钟级」——这是一个非线性的临界点。达到这个密度后，该城市对 Agent 开发者的吸引力会急剧上升，需求侧自然加速。

## 十、与中国版的互补关系

国际版和中国版不是替代关系，而是两套互补的基础设施：

| 维度 | 国际版 | 中国版 |
|------|--------|--------|
| 支付 | USDC + 法币通道 | 支付宝 / 微信 / 灵活用工发票 |
| 执行者入口 | Telegram Bot | 微信小程序 |
| 合规路径 | 独立承包商（1099 模式）| 灵活用工三方协议 |
