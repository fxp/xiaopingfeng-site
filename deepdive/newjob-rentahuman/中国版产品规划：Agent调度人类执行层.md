# 中国版产品规划：Agent 调度人类执行层（ToB SaaS）

> **产品定位**：为中国企业级 AI Agent 提供"人类即 API"的执行基础设施——当 Agent 遭遇物理世界边界时，调用真实人类完成任务，全链路自动化结算与合规。
>
> **文档类型**：探索期产品规划（PRD 级）  
> **适用阶段**：0→1 方向验证  
> **撰写日期**：2026-04-30

---

## 一、为什么中国需要不一样的产品形态

RentAHuman 的原型证明了需求存在，但它的整个设计假设——加密支付、无 KYC、匿名委托——在中国市场是负债，不是资产。

中国的机会恰恰在于反向：**用最严格的合规换取最大的市场空间**。滴滴、美团的众包模式证明了「平台调度分散劳动力」在中国可以做到数百亿规模；AI Agent 的兴起是在这个成熟基础上增加了一个新的委托主体。

核心判断：**中国版产品的差异化不在于「AI 雇人」这个概念本身，而在于把这个概念做成合规、可审计、企业可集成的 B 端基础设施。**

## 二、市场机会

### 2.1 为什么现在

三个条件在 2026 年同时成熟：

**AI Agent 落地加速**：国内头部企业（制造、零售、金融、物流）正在规模化部署 AI Agent，但普遍卡在同一个瓶颈——Agent 触达不了物理世界。质检 Agent 需要有人去现场拍照、合规 Agent 需要有人去签字、营销 Agent 需要有人去做实地调研。这类需求真实存在，但没有标准化的接入层。

**灵活用工市场成熟**：中国灵活用工平台（美团众包、滴滴众包、斗米、快手招聘）已经培养了数千万注册零工用户。人力池是现成的，缺的是「可以被程序化调用的接入接口」。

**国内大模型生态完备**：Qwen、DeepSeek、Kimi、文心一言均已支持 Function Calling / Tool Use，MCP 协议在开发者社区正在快速普及。Agent 调用外部工具的技术链路已经成熟，缺的是「人类执行层」这个工具。

### 2.2 市场规模估算

| 层次 | 数据 |
|------|------|
| 中国灵活用工市场规模（2025）| 约 1.3 万亿人民币 |
| 企业 AI Agent 采购意愿（Gartner 2026 预测）| 年增速 40%+ |
| 当前国内「外勤执行类 SaaS」市场头部玩家年营收 | 5-20 亿级 |
| 目标可服务市场（初期 2 年）| 企业客户 AI Agent 外勤任务外包，预估 TAM 50-200 亿 |

## 三、目标客户与核心场景

### 3.1 理想客户画像（ICP）

**第一优先级：有 AI Agent 落地但卡在物理执行的企业**

- 制造业质检团队：Agent 已跑通视觉分析，但抽样送检、现场核验还需要人
- 金融机构贷前调查：信贷 Agent 已完成数据核查，但需要实地走访核验经营状况
- 快消/零售巡检：AI 已分析门店数据异常，需要有人现场拍照确认
- 保险理赔：AI 已判断理赔合理性，需要定点勘察员现场取证
- 法律/公证服务：合规 Agent 完成文件起草，需要线下见证、签字、公证

**第二优先级：外包执行需求大、管理成本高的业务场景**

- 市场调研公司：用 AI 设计问卷，用平台调度访谈员
- 跨境电商选品：AI 分析趋势后，需要人去线下市场采样拍照
- MCN/媒体内容：AI 规划内容脚本，需要实地探店/街访/录制

### 3.2 核心任务类型矩阵

| 任务类型 | 频率 | 单价区间 | 技能要求 |
|---------|------|---------|---------|
| 现场拍照/视频采集 | 高频 | 50-200 元/次 | 无门槛 |
| 实地走访核验 | 中频 | 150-500 元/次 | 基础沟通 |
| 问卷访谈 | 中频 | 100-300 元/小时 | 普通话标准 |
| 签字见证/文件递送 | 低频 | 200-800 元/次 | 身份核验 |
| 专业实地调研 | 低频 | 500-2000 元/次 | 行业背景 |
| 代理采购/线下验货 | 中频 | 任务金额的 5-10% | 无门槛 |

## 四、产品架构

整个产品分为三层，对应三类用户：

```
┌─────────────────────────────────────────────────────┐
│                    企业 AI Agent                     │  ← 委托方
│         （通过 API / SDK / MCP 接入）               │
└─────────────────┬───────────────────────────────────┘
                  │ 任务委托
┌─────────────────▼───────────────────────────────────┐
│              执行层调度平台（核心产品）               │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌────────┐ │
│  │ 任务引擎 │ │ 匹配引擎 │ │ 验证引擎 │ │结算引擎│ │
│  └──────────┘ └──────────┘ └──────────┘ └────────┘ │
│  ┌──────────────────────────────────────────────┐   │
│  │        风控 · 合规 · 审计日志               │   │
│  └──────────────────────────────────────────────┘   │
└─────────────────┬───────────────────────────────────┘
                  │ 任务分发
┌─────────────────▼───────────────────────────────────┐
│               人类执行者（执行方）                   │
│       （通过微信小程序 / APP 接收和完成任务）        │
└─────────────────────────────────────────────────────┘
```

### 4.1 核心模块说明

**① 任务引擎（Task Engine）**

- 接收 Agent 发来的结构化任务请求（JSON Schema 定义）
- 支持任务类型：一次性任务、周期性任务、条件触发任务
- 任务参数：地理位置（可选）、截止时间、完成证明要求（照片/视频/文字/电子签）、预算上限
- 任务生命周期管理：待匹配 → 已接单 → 执行中 → 待验证 → 已完成 / 超时 / 争议

**② 匹配引擎（Matching Engine）**

- 执行者标签体系：地理位置、技能标签、历史评分、认证等级（实名 / KYC / 专业认证）
- 匹配策略：LBS 优先 → 评分排序 → 价格过滤
- 支持两种模式：
  - **直接指派**（Agent 指定条件，平台自动派单）
  - **悬赏模式**（任务挂出，执行者主动抢单）
- 超时自动重派机制

**③ 验证引擎（Verification Engine）**

- 完成证明接收：图片/视频（带 GPS 水印、时间戳、防篡改）
- AI 预审：自动校验位置信息、图像内容与任务要求的符合度
- 争议处理流程：人工复核队列（初期由运营团队处理）
- 验证通过后自动触发结算

**④ 结算引擎（Settlement Engine）**

- 对接企业侧：支付宝企业付款、微信企业付款、银行对公转账
- 对接执行者侧：支付宝/微信个人收款
- **合规结算路径**（关键）：
  - 灵活用工综合服务平台模式（三方协议：委托企业 + 灵活用工平台 + 个人，税务代扣代缴）
  - 执行者无需自行处理税务，企业可拿合法发票
- 资金托管：任务发起时预授权冻结，完成后释放；超时自动退款

## 五、技术选型

### 5.1 Agent 接入层（最核心的差异化）

**MCP Server（首选集成方式）**

国内主要 Agent 框架和 IDE 均支持 MCP 协议。提供标准 MCP Server，执行者工具集如下：

```json
{
  "tools": [
    "create_task",          // 创建任务
    "get_task_status",      // 查询任务状态
    "list_available_humans",// 浏览执行者列表
    "cancel_task",          // 取消任务
    "get_completion_proof"  // 获取完成证明（含 URL）
  ]
}
```

**REST API（标准 HTTP 接口）**

为无法使用 MCP 的场景提供等价 API，支持 Webhook 回调（任务状态变更时主动推送）。

**SDK（Python / TypeScript 优先）**

```python
from meatspace import MeatspaceClient

client = MeatspaceClient(api_key="...")
task = client.tasks.create(
    type="field_photo",
    location={"city": "上海", "address": "徐汇区天钥桥路 100 号"},
    instruction="拍摄门店正面照，要求完整展示招牌和橱窗，共 3 张",
    proof_type=["gps_photo"],
    budget_cny=80,
    deadline_minutes=120
)
# task.id, task.status, task.webhook_url
```

**国内大模型兼容性**

| 大模型 | Tool Use 支持 | MCP 支持 | 集成优先级 |
|--------|-------------|---------|----------|
| Qwen（通义千问）| ✅ | ✅ | P0 |
| DeepSeek | ✅ | ✅ | P0 |
| Kimi（月之暗面）| ✅ | 路线图中 | P1 |
| 文心一言 | ✅ | 路线图中 | P1 |
| Claude（通过 API）| ✅ | ✅ | P1 |

### 5.2 执行者端（微信小程序优先）

- 中国执行者的核心触点是微信，APP 冷启动成本过高
- 小程序支持：任务接收通知、地图导航、GPS 打卡、拍照上传（带水印）、收款查询
- 实名认证通过微信实名体系（无需额外 KYC 成本）

### 5.3 基础设施选型

| 组件 | 选型 | 理由 |
|------|------|------|
| 云服务 | 阿里云 / 腾讯云 | 数据本地化合规 |
| 地图 / LBS | 高德 API | 覆盖最全，企业 API 稳定 |
| 支付 | 支付宝企业 + 微信支付 | 国内覆盖率 |
| 实名认证 | 微信实名 / 阿里实人认证 | 成本低，合规性高 |
| 消息通知 | 微信服务通知 + 短信 | 触达率高 |
| 存储（证明文件）| OSS（带不可篡改日志）| 司法存证级别 |
| 数据库 | PostgreSQL + Redis | 标准选型 |

## 六、合规设计（中国市场最关键的护城河）

这是中国版产品与 RentAHuman 最本质的差异，也是 ToB 市场的核心门槛。

### 6.1 灵活用工合规路径

**问题**：AI Agent 委托个人完成任务，产生劳动报酬，必须解决税务合规问题，否则企业无法入账，执行者面临偷税风险。

**解决方案**：接入灵活用工综合服务平台（合法的三方结算模式）

```
企业客户 → 与灵活用工平台签订服务协议 → 平台与执行者签订用工协议
      ↓                                          ↓
  支付含税服务费                            获得合规劳务收入（个税代扣）
      ↓                                          ↓
  取得增值税发票                            无需自行报税
```

国内已有成熟的灵活用工 SaaS 服务商（如薪云、薪人薪事、易路、壹号人事），可作为结算通道合作方，无需自建资质。

**核心合规要点**：
- 执行者须完成实名认证（身份证 + 人脸识别）
- 单次结算超过 800 元须代扣个税
- 企业端取得正规增值税发票（6% 现代服务业）
- 平台不构成劳动关系（非全日制、非固定用工）

### 6.2 平台资质需求

| 资质 | 是否必须 | 获取难度 |
|------|---------|---------|
| 增值电信业务许可证（ICP）| 必须 | 低，3-6 个月 |
| 网络信息内容服务备案 | 必须 | 低，1 个月 |
| 人力资源服务许可证 | 建议 | 中，3-6 个月 |
| 劳务派遣经营许可 | 初期不需要（灵活用工模式规避）| — |
| 支付业务：商户号即可 | 无需支付牌照 | 低 |

### 6.3 AI 委托任务的风控机制

这是监管层面最敏感的创新点，需要主动设计：

**任务内容审核**（不能完全依赖 AI 委托方的自律）：
- 任务标题 + 描述经过大模型内容安全审核（违禁词、高风险指令识别）
- 任务类型白名单制度（初期限定 10-20 个预定义任务类型，拒绝自由文本的高风险任务）
- 涉及人脸采集、私人住宅进入等特殊类型须人工预审

**委托方企业资质审核**：
- 企业客户须完成工商信息核验（通过天眼查 / 企查查 API）
- API Key 与企业主体绑定，任务日志全量留存 180 天
- 高风险任务类型须企业签署额外承诺协议

**执行者保护机制**：
- 任务发起时展示完整指令，执行者有权拒绝
- 不允许 AI 在任务执行过程中修改指令（指令锁定）
- 争议 SLA：24 小时内人工介入

## 七、商业模式

### 7.1 收入结构

**主收入：平台服务费（抽成模式）**

- 向委托企业按任务结算金额收取 15-25% 平台服务费
- 执行者拿到全额（税后）——这是执行者侧的拉新关键

**次收入：企业订阅（API 额度包）**

| 套餐 | 月费 | 包含内容 |
|------|------|---------|
| 基础版 | 2,000 元/月 | API 接入 + 1,000 次任务/月 + 标准 SLA |
| 专业版 | 8,000 元/月 | + 优先匹配 + 自定义任务类型 + 专属客户成功 |
| 企业版 | 定制 | 私有化部署 + 专属执行者池 + 合规报告 |

**增值收入：合规服务包**

- 灵活用工发票服务：任务金额的 1-3%（含税务代扣代缴）
- 司法存证服务：0.5 元/份证明文件（区块链存证）
- 执行者专业认证服务：向有认证需求的执行者收取年费

### 7.2 单位经济模型

以一个「现场拍照」任务为例（最高频场景）：

```
企业委托价：100 元
├── 执行者所得：75 元（税后）
├── 平台服务费：20 元（20%）
│   ├── 结算通道成本：3 元
│   ├── 匹配/技术成本：2 元
│   └── 毛利：15 元（毛利率 15%）
└── 灵活用工发票：5 元（企业额外支付）
```

**达到盈亏平衡的关键指标**：
- 月任务量：约 5,000 单（假设平均单价 200 元，20% 抽成）
- 对应月 GMV：100 万元，平台收入 20 万元
- 固定成本（团队 5 人，服务器，合规成本）：约 15-20 万元/月

## 八、冷启动策略

双边市场冷启动是这类平台最难的问题。解法是：**先锁定供给，后释放需求；先垂直深，后横向扩。**

### 8.1 执行者侧（供给方）冷启动

**第一批执行者：不从零招募，从现有平台导流**

- 与 1-2 家灵活用工平台（斗米、找活、人瑞人才）谈数据合作，导入其已有实名认证用户池
- 与本地高校就业中心合作（大学生是最容易激活的零工群体）
- 初期目标：3 个城市（上海/北京/广州）各 500 名认证执行者上线

**第一批执行者激活**：
- 新执行者完成首单奖励 50 元
- 前 3 个月平台零抽成（执行者拿 100% 任务金额）
- 微信群运营：城市执行者群，运营人员每天发任务通知

### 8.2 企业侧（需求方）冷启动

**策略：用免费试用换第一批真实任务数据**

- 第一步：锁定 5-10 家有明确 AI Agent 外勤需求的企业（制造业质检、连锁零售巡检）
- 与这些企业共同「定制任务类型」——他们提需求，我们提技术方案
- 前 3 个月：免平台服务费，只收灵活用工发票成本
- 目标：跑通 100 个真实任务，积累案例和证明数据

**销售路径**：不走大客户直销，先从「AI Agent 开发者社区」切入

- 在 GitHub、Discord、微信技术群中推广 MCP SDK
- 让 Agent 开发者在自己的项目中集成，把平台变成开发者工具而非传统 SaaS
- 开发者用了 → 汇报给企业决策层 → 企业买单

### 8.3 冷启动时间线

| 时间 | 里程碑 | 关键行动 |
|------|--------|---------|
| M1-2 | 技术 MVP 上线 | MCP Server + REST API + 微信小程序基础版 |
| M3 | 第一批闭环 | 5 家种子企业，100 个真实任务完成 |
| M4-6 | 扩城市 | 新增 3 个城市执行者池，任务类型扩展到 15 个 |
| M6 | 开始商业化 | 开始收取平台服务费，目标月 GMV 50 万元 |
| M9-12 | 产品标准化 | 完善自动匹配、AI 验证、争议处理，减少人工干预 |
| M18 | 规模化 | 月 GMV 500 万元，覆盖 10 个城市，50+ 企业客户 |

## 九、竞争分析与护城河

### 9.1 现有玩家的局限

| 现有平台 | 能力 | 缺口 |
|---------|------|------|
| 美团众包 / 滴滴众包 | 大规模 LBS 调度、已有执行者池 | 无 API 接口供 Agent 调用，任务类型固定 |
| 猪八戒 / 威客平台 | 技能外包 | 以人类为委托方，无 Agent-native 接入 |
| 企业 HR SaaS（北森等）| 人员管理 | 不支持灵活任务调度 |
| RPA 工具（影刀等）| 数字自动化 | 到物理世界边界就停了 |

**核心结论**：没有现成的竞品在做「AI Agent 调度人类执行者」这件事，这是一个真正的蓝海窗口，但窗口期可能不长——一旦美团/阿里意识到这个方向，他们的执行者池和支付基础设施是降维打击。

### 9.2 如何建立护城河（时间窗口内）

**护城河 1：Agent 开发者生态绑定**  
成为国内 AI Agent 开发者默认的「人类执行层工具」。SDK 越早进入各大 Agent 框架的示例代码和文档，迁移成本就越高。这类基础设施产品有显著的路径依赖效应。

**护城河 2：垂直场景壁垒**  
金融贷前调查、保险实地核验、连锁零售巡检——这些场景有强合规需求，一旦跑通了与行业监管兼容的任务流程，新进入者需要重新谈判合规路径。

**护城河 3：执行者质量体系**  
高质量、可信赖的执行者池需要时间积累。KYC 认证、历史评分、专业认证标签——这是一个随时间积累的数据飞轮，早期竞争者有先发优势。

**护城河 4：合规成本是进入壁垒**  
中国市场的灵活用工合规、数据安全法合规，对外资或新入场玩家构成实际门槛。早期把合规做扎实的玩家，在规模化时能快速取得企业信任。

## 十、关键风险与应对
