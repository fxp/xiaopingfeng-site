## 3. 帧格式

```
请求 (上位机 → 下位机)：
+------+------+--------+-----+------+--------+--------+
| 0xEE | 0x01 | addr×4 | cmd | len  | data×N | crc×2  |
+------+------+--------+-----+------+--------+--------+

响应 / 主动上报 (下位机 → 上位机)：
+------+------+--------+-----+------+--------+--------+
| 0xFF | 0x01 | addr×4 | cmd | len  | data×N | crc×2  |
+------+------+--------+-----+------+--------+--------+
```

`len` 为 2 字节大端，**仅指 data 部分长度**。`crc` 为 CRC16-XMODEM 大端。

## 5. 发现的协议层问题

### #1 — 文档写 crc16，设备根本不校验
CSV 只写"crc16"。穷举 18 种常见变种后发现：**设备不校验 CRC，填任意值均被接受**。

**测试方法**：发送 CRC 字段故意填错的帧，设备正常应答。进一步用已知帧遍历 18 种变种均无一匹配，确认设备无校验行为。

### #2 — 设备地址默认值不对
CSV 示例使用 `0x00000001`，本机拨码实测是 `0x00000000`。所有库代码必须把地址做成可配置项。

### #3 — `0x2B` 应答结构跟 CSV 不一致
CSV 写：`1B 层数 + N B 每层数 + 4B*层数 (Y步数) + 4B*总货道 (X步数)` = 180 字节。
**实际** 152 字节，结构是：

```
[1B 总层数=7][7B 每层数 3,5,5,5,6,6,6][4B × 36 lanes 步数]
```

没有"每层 Y 步数"块。每个 lane 一个 4 字节整数（看数值像 X 位置，但无标识）。

### #4 — `0x24` 应答字节 1 不是状态码
CSV 写 byte[1] 为扫描状态（0/0xFF），**实际**返回层数（如 7）。

### #5 — `0x24` 实际耗时远超预期
扫描全部 36 货道实测 **202 秒**。CSV 没说，最初库默认 60s timeout 必然失败（且失败后回包会丢，需 0x2A 复位）。

### #6 — `0x29` echo 字段语义不明
- lane=`0x0001` → echo `00 01`
- lane=`0x0064` → echo `00 00`
- lane=`0x0065` → echo `00 01`
- lane=`0x00FF` → 无应答
- lane=`0x0101`+ → 无应答

前 2 字节看起来像某种 mod 映射，但**只有最后 1 字节状态可靠**。

### #7 — `0x35 type=1` 异常应答 cmd 字段
请求 `0x35 + payload=0x01`，应答 `cmd=0x36`。其它 type（0/2/3）正常。判定为 firmware 没实现横红外，应答用了占位 cmd。

### #8 — `0x04` 永远报"无电机"
请求 `0x04` 不论 lane 字段填什么（`0xFF` 跳过 / `100` 真 lane），都返回：
```
device_status=0x00 has_goods=0 can_dispense=no reason=0x02 (无电机)
```
但 `0x28` 直接对 lane 100 出货实际成功。**`0x04` 不可作为出货预检使用**。

### #9 — 长操作期间 `reset_input_buffer()` 会丢应答
`0x24` 这类 200s 才回的指令，期间若用户发别的查询，库的 `reset_input_buffer()` 会冲掉迟到的应答。需要"被动监听不主动 flush"模式。

### #10 — 5 条指令 firmware 不实现
`0x3D` / `0x23` / `0xBC` / `0x2C` / `0x35 type=1`。判定依据见 §4.3。

## 8. 货道编号方案

CSV 没明确说，实测情况：

- **0x28 出货指令**用平面十进制编号，实测 lane=100（`0x0064`）有效并能出货
- **0x29 查电机**接受相同编号空间，但前 2 字节 echo 含义不明
- **0x2B 全步数**返回 36 个 4-byte 值，与 36 个 lane 数量匹配，但**索引到 lane 编号的映射未知**

实务上：要使用具体 lane 必须知道该机器的 lane 命名表（通常贴在机器内或由商家提供）。

## 9. 生产代码建议

1. **出货流程**：`0x28` 直接发，**不要**用 `0x04` 预检；通过 `0xE1` 主动上报判断进度；终态查 `0x30`
2. **错误恢复**：用 `0x2A`，**不要**用 `0x23`
3. **状态轮询**：`0x05` 即足够，特殊情况补 `0x39` 微动
4. **启动初始化**：上电后跑一次 `0x24`（timeout ≥ 300s）建立货道映射
5. **CRC**：设备不校验，填 `0x0000` 即可；若需防御性实现，可用 XMODEM 大端
6. **超时策略**：长操作（0x24, 0x28, 0x65）期间不要 flush input buffer
7. **firmware 黑名单**：避免依赖 `0x3D` / `0x23` / `0xBC` / `0x2C`
